# Paythor SanalPosPro — Development Phases

This document summarizes the 3-phase development process of the `Paythor_SanalPosPro` and `Eticsoft_PaythorClient` modules, along with their technical details.
 
---

## Module Architecture

```
Eticsoft_PaythorClient          Paythor_SanalPosPro
  (API Client)                    (Payment Integration)
  ├─ SDK logic                    ├─ Gateway API (Phase 2)
  ├─ HTTP requests                ├─ Admin Config (Phase 1)
  └─ Paythor API comms            ├─ Checkout UI (Phase 3)
                                  └─ Observer / Handler
```

The `Paythor_SanalPosPro` module depends on `Eticsoft_PaythorClient` and utilizes its SDK/API layer.

---

## Phase 1 — Skeleton, Dependencies, and Admin Configuration

### Objective
To build the module skeleton, register it with Magento, and create the admin panel configuration alongside the database schema.

### Created / Updated Files

| File | Description |
|---|---|
| `etc/module.xml` | Module declaration. `setup_version` is not used (Declarative Schema). Dependencies defined with `<sequence>` include `Eticsoft_PaythorClient`, `Magento_Payment`, `Magento_Sales`, `Magento_Checkout`, and `Magento_Store`. |
| `etc/adminhtml/system.xml` | Admin panel settings: Active status, title, payment action (authorize/capture), test mode, API Key, API Secret (obscure), Webhook Secret (obscure), currency, and sort order. |
| `etc/config.xml` | Default values: `model` → `PaythorSanalPosProFacade`, `can_authorize`, `can_capture`, `can_refund`, `can_void`, `active`, `title`, `sort_order`, API field defaults. |
| `etc/acl.xml` | Access Control List (ACL): Resource definitions for `Paythor_SanalPosPro::config`, `::paythor`, `::connect`, and `::transactions`. |
| `etc/db_schema.xml` | `paythor_transaction_log` table: `entity_id`, `order_id`, `quote_id`, `paythor_transaction_id`, `paythor_process_id`, `status`, `amount`, `currency`, `request_payload`, `response_payload`, `created_at`, `updated_at`. Foreign key: `sales_order.entity_id`. |
| `etc/db_schema_whitelist.json` | Declarative Schema whitelist file. |

### Technical Decisions
- The `setup_version` attribute was intentionally omitted per `.cursorrules` — all DB changes are managed via `db_schema.xml`.
- The `api_secret` and `webhook_secret` fields are defined as `obscure` (encrypted storage).
- Sensitive data fields (`api_key`, `api_secret`, `webhook_secret`, `sandbox_mode`) are marked as `environment` in `di.xml` via `TypePool` → meant to be stored in `env.php`, not DB.

### Validation
```bash

bin/magento setup:di:compile
```

## Phase 2 — Gateway Backend (Payment Provider Gateway API)

### Objective
To implement the `authorize`, `capture`, `refund`, and `void` application flow in accordance with Magento 2's Commerce Payment Provider Gateway architecture. Wrapping the existing `PaythorAdapter` SDK with the Gateway API layer.

### Architecture Diagram

```
placeOrder()
  │
  ▼
CommandPool (di.xml)
  ├─ authorize → GatewayCommand
  │    ├─ RequestBuilder (AuthorizationDataBuilder + PaymentDataBuilder)
  │    ├─ TransferFactory
  │    ├─ PaythorGatewayClient → PaythorAdapter.createPaymentFromGateway()
  │    ├─ GeneralResponseValidator
  │    └─ HandlerChain (TransactionIdHandler + PaymentDetailsHandler)
  │
  ├─ capture → GatewayCommand
  │    ├─ CaptureDataBuilder + PaymentDataBuilder
  │    ├─ PaythorGatewayClient → PaythorAdapter.captureFromGateway()
  │    └─ ...
  │
  ├─ refund → GatewayCommand
  │    ├─ RefundDataBuilder + PaymentDataBuilder
  │    ├─ PaythorGatewayClient → PaythorAdapter.refundFromGateway()
  │    └─ ...
  │
  └─ void → GatewayCommand
       ├─ VoidDataBuilder + PaymentDataBuilder
       ├─ PaythorGatewayClient → PaythorAdapter.voidFromGateway()
       └─ ...
```

### Created / Updated Files

| File | Description |
|---|---|
| **Gateway/Http/TransferFactory.php** | `TransferFactoryInterface` implementation. Transforms Request data into a `TransferInterface` object. |
| **Gateway/Http/Client/PaythorGatewayClient.php** | `ClientInterface` implementation. Invokes the respective adapter bridge method in `PaythorAdapter` according to the transaction type (`authorize`, `capture`, `refund`, `void`). |
| **Gateway/Request/PaymentDataBuilder.php** | Common payment data builder: `order_id`, `amount`, `currency`, `customer_email`. Implements `BuilderInterface`. |
| **Gateway/Request/AuthorizationDataBuilder.php** | For `action: authorize`, maps `paythor_token` and card details. |
| **Gateway/Request/CaptureDataBuilder.php** | For `action: capture`, includes `transaction_id` (from previous authorization). |
| **Gateway/Request/RefundDataBuilder.php** | For `action: refund`, includes `transaction_id` and `refund_amount`. |
| **Gateway/Request/VoidDataBuilder.php** | For `action: void`, includes `transaction_id`. |
| **Gateway/Response/TransactionIdHandler.php** | Parses `transaction_id` from API responses and securely stores it using `Payment->setTransactionId()`. |
| **Gateway/Response/PaymentDetailsHandler.php** | Stores additional response details (`status`, `process_id`, etc.) via `Payment->setAdditionalInformation()`. |
| **Gateway/Validator/GeneralResponseValidator.php** | Extension of `AbstractValidator`. Validates the HTTP status code, `success` flag, and handles Paythor-specific error codes. |
| **Observer/DataAssignObserver.php** | Extension of `AbstractDataAssignObserver`. Maps `additional_data` from the frontend to `PaymentInfo.additionalInformation`. Allowed keys: `paythor_token`, `payment_method`. |
| **Model/Ui/ConfigProvider.php** | `ConfigProviderInterface` implementation. Provodes `isActive`, `title`, `isSandbox`, and `paymentAction` variables to `window.checkoutConfig.payment.paythor_sanalpospro`. |
| **Model/Api/PaythorAdapter.php** | Added bridge methods to the existing adapter: `createPaymentFromGateway()`, `captureFromGateway()`, `refundFromGateway()`, `voidFromGateway()`, and `normalizeApiResponse()`. |
| **etc/di.xml** | Complete Gateway Facade configuration mapping: `PaythorSanalPosProFacade` (VirtualType → `Magento\Payment\Model\Method\Adapter`), `CommandPool`, `ValueHandlerPool`, `ValidatorPool`, `CountryValidator`, Builders composites, Handler chains, and Logger DI. Enforces sensitive data constraints using `TypePool`. |
| **etc/events.xml** | Registers `payment_method_assign_data_paythor_sanalpospro` observer → `DataAssignObserver`. |
| **etc/frontend/di.xml** | Registers `ConfigProvider` to `Magento\Checkout\Model\CompositeConfigProvider`. |
| **etc/config.xml** | Updated the `model` attribute value to the `PaythorSanalPosProFacade` VirtualType. |

### Technical Decisions
- Preserved existing `PaythorAdapter` SDK logic, while successfully wrapping it using the Gateway API layer through the **bridge pattern**. `PaythorGatewayClient` does not perform direct HTTP calls manually; it transfers the responsibility completely to the adapter.
- `ObjectManager` was not used anywhere directly — dependencies were strictly handled via Constructor Injection utilizing DI mapping constructs.
- `GeneralResponseValidator` manages both regular HTTP exceptions and designated Paythor specific error payloads simultaneously.
- Handler classes are streamlined specifically for parameter assignments and do deliberately not contain any core business logic rule processing (a strict `.cursorrules` mandate).

## Phase 3 — Frontend Checkout (Knockout.js / UI Components)

### Objective
Ensure that the payment method is gracefully rendered as a proper Knockout.js UI Component entirely on the Checkout page, complete the `getData()` payload propagation correctly via the Gateway API, construct a robust model-based secure dialog window frame overlay, and inject modern CSS aesthetic styling overrides accordingly.

### Payment UX Flow

```
1. Customer clicks "Place Order"
   │
2. Magento's setPaymentInformationAction() executes → backend receives additional_data payload
   │   ├─ paythor_token (if available)
   │   └─ payment_method: 'credit_card'
   │
3. AJAX POST → proceeds correctly to paythor/payment/create 
   │   → Backend CreateController subsequently returns dynamic iframe HTML content wrapper
   │
4. The frontend Modal gracefully opens up → The dedicated Paythor iframe visualization runs flawlessly
   │   → Client subsequently keys their private card entries into Paythor's highly secured PCI compliant component space form securely
   │
5a. postMessage execution propagates (from Paythor iframe container directly → to the main parent DOM Window context instance)
   │   ├─ Status Success: Execute _sendConfirmRequest() operation → generates backend Order sequence safely → executes native system level redirect mechanism precisely finally
   │   └─ Status Error Failure: Modal window unceremoniously closes natively, prominent alert text generated logically, ongoing Cart session intentionally retained properly.
   │
5b. Execution fallback utilizing standard Callback bridge pattern mechanism (Paythor redirects seamlessly → natively toward Callback.php handler layer → fires secure isolated postMessage event directly)
       └─ Mirrors and perfectly aligns to the precise identically modeled behavior sequence flow natively as noted identically directly above completely.
```
